## Phase 2 — Architecture / Duplication

**Fixed (production bug):**
- **Duplicated function `elegirFotoProd`** declared twice in the frontend for two
  different contexts (product-form photo → `_prodFoto`; purchase-product photo →
  `_cprFoto`). The second declaration silently overrode the first, so the product
  form's photo input invoked the **wrong** handler and product photos broke.
  → Renamed the product-form handler to `elegirFotoProdForm` and repointed its
  input. Added E2E regression asserting both handlers exist and are distinct.

**Removed (confirmed dead code, zero references):**
- `dashboardHTML()` — superseded by the inline render in `loadHome()`.
- `qhaHomeHTML()` — superseded home "qué hacer ahora" renderer.
- `puedeVer()`, `modulosVisibles()` — legacy role helpers (role gating now lives in
  `auth.requerido` server-side + `modulosMeta`/`CATS` client-side).
- Orphaned constants `MODULOS_EQUIPO`, `MODULOS_ADMIN`.

**Flagged (dead-but-feature-adjacent — not removed to avoid deleting intended work):**
- Camera cluster `abrirCamara()` / `capturarFoto()` (unreachable: no caller invokes
  `abrirCamara`). Self-contained; safe to remove or wire up.
- `confirmarRcp()` + backend `POST /recepciones/:id/confirmar` — reception-confirm
  flow present but not wired to any button. Decide: wire it up or remove both.

**Not a defect:** money math like `disponibilidad * coste_medio` appears in a few
modules but each composes `costing.js` values; consolidating the last inline
valuations into `costing` helpers is a low-risk future tidy (tracked below).

## Phase 5 — Performance

- The executive dashboard previously scanned `ventas` ~14× per load; already
  optimized to compute `beneficio`/`costeMedioDiario` once and memoize per period.
- No N+1 remote calls (JSON/PG reads are in-memory cache after load).
- Future: cache `indiceMaterias` per request for large `ventas` histories; add PG
  indexes when the SQL path scales (see Engineering Volume IV · 35 Performance).

## Phase 6 — Security

- **No hardcoded secrets** in code (grep clean). JWT secret **required in
  production** (`REQUIRE_JWT_SECRET`), random ephemeral fallback otherwise.
- **PINs hashed** (scrypt + salt), never stored in clear; brute-force lockout after
  5 attempts (429).
- **Role gating:** any route segment outside `EQUIPO_ALLOWED` is admin-only by
  default; all financial segments are excluded + reaffirmed by an explicit guard.
- **XSS:** `esc()` applied on user-facing sinks (167 `esc()` vs 54 `innerHTML`).
- **Audit trail** present for critical actions with user + local_id.
- Only 4 `console.log` in backend (low noise).

## Phase 7 — Scalability

- `local_id` already exists on user sessions; entities are ready to carry it.
- Persistence abstraction supports PostgreSQL for multi-tenant scale.
- Financial entities modeled for multi-location/franchise/enterprise from day one.
- Follow-up: formal `local_id` propagation across all entities (Future Volume V · 40).

## Phase 8 — Complexity Removal

Duplicate function collision removed; dead renderers/constants removed; owner
dashboards already unified. No duplicate money engines, inventory engines or
business-rule implementations found.

## Phase 9 — Standardization (observations)

- Routes follow a consistent shape (`camposDe` validators, guard, `store` CRUD,
  `flush()`); financial routes share `_guard.soloAdmin`.
- Frontend uses dynamic dispatch `irA_${id}()` — powerful but defeats static
  dead-code analysis; recommend a small central route registry as the SPA grows.

## Phase 10 — Future-Readiness
